Basic linux pthread usage and principle

Linux pthread

POSIX 线程库（<pthread.h>）是在 Linux 下为 C/C++ 提供的 POSIX 标准线程库，其提供了创建线程、销毁线程、操纵线程、锁等一系列 API。

Thread Basics

  • 线程操作包括线程的创建（creation）、终止（termination）、同步（synchronization, join, blocking）、调度（scheduling）、数据管理（data management）和进程/线程间交互（process interaction）

  • 线程既不维护它所创建的所有线程的列表，也不知道它是被谁所创建的

  • 一个进程内的所有线程共享相同的地址空间

    具体的实现方式在不同平台上有很大的区别，Linux 下进程和线程的结构体其实是一样的，但是在创建过程中所使用的参数有较大区别。

  • 同一个进程内的所有线程共享

    • 进程指令（注意，这里共享的是指令本身，而不是指令在执行过程中的寄存器）
    • 大多数的数据
    • 文件描述符（已经被打开的）
    • 信号和信号处理器（signals and signal handlers）
    • 当先的工作目录
    • 用户和用户组ID

  • 每个线程维护其独有的

    • 线程 ID （Thread ID）
    • 寄存器，堆栈指针
    • 堆栈局部变量，返回地址
    • 信号掩码（signal mask）
    • 优先级（priority）
    • 返回值

Thread Synchronization

pthread 库提供了三种线程同步方法，分别是：

  • mutex（注：在内核中还有更多类型的锁，如 rw_lock、spinlock、rcu 等，这里的 mutex 是 pthread 库提供的用户态互斥锁）
  • joins
  • condition variables

Mutex

Mutex（互斥锁）用于防止由条件竞争（race condition）引发的数据不一致。在任何时候如果有多个线程试图访问/修改一个共享内存区域，都应该使用 mutex 对其进行保护。和信号量不同，Mutex 不能在进程间作用。

Condition Variables

条件变量是类型为 pthread_cond_t 结构的一类变量，条件变量的机制允许线程挂起并等待直到某些条件为真之后再继续运行。条件变量必须与一个 mutex 互斥锁相关联。

但是看到这里我依然存在疑问，即 condition variables 中它的 lock 是怎么运作的，这个 mutex lock 的获取与释放顺序显得尤为奇怪，以及为什么会需要这样的一个 lock 呢？

首先对 pthread_cond_wait 函数进行简单的理解，该函数在进入等待（或者说被调度）之前会释放所持有的 mutex 锁，而当其被唤醒之前，它会重新尝试获取锁，上述的这些内容都是隐式执行的。因此如果 signalling threads（发信号的线程）此时持有锁，那么等待线程将不会被唤醒，直到 signalling threads 释放锁。

现在，我又引入了以下几个问题：

  • 为什么等待线程要使用 while 而不是 if ？
  • 为什么要在 pthread_cond_wait 内使用互斥锁 mutex？

More details

首先，这里再强调下，条件变量本身不需要我们再用锁进行保护，使用条件变量的过程中，对我们真正有意义的，是全局变量“条件”，而不是条件变量，因此所有的 mutex 锁的目录都是为了更好的保护全局“条件”的一致性。而条件变量本身，如果它在实现过程中需要锁，那么它在内部会自己维护锁，不需要使用者再去传入。

这里的条件变量，我们可以将其简单的理解为一个队列，每一个等待条件变量的线程都会将自己插入到该队列的末尾中去。

那么首先来解决第一个问题，即为什么使用 while 判断而不是 if，经过了解，这里是为了避免 虚假唤醒（Spurious wakeup），借用 wiki 的内容：

A spurious wakeup happens when a thread wakes up from waiting on a condition variable that’s been signaled, only to discover that the condition it was waiting for isn’t satisfied. It’s called spurious because the thread has seemingly been awakened for no reason. But spurious wakeups don’t happen for no reason: they usually happen because, in between the time when the condition variable was signaled and when the waiting thread finally ran, another thread ran and changed the condition. There was a race condition between the threads, with the typical result that sometimes, the thread waking up on the condition variable runs first, winning the race, and sometimes it runs second, losing the race.

大致来说，就是线程在不该被唤醒的时候唤醒了，那么为什么会发生虚假唤醒的情况呢？

这里首先回答第二个问题，即为什么要在 pthread_cond_wait 函数中引入 mutex 锁。我们知道，这里的 mutex 锁是来保护“全局条件”的，因此在等待线程判断条件之前，就需要获取锁，当获取锁之后，等待线程会判断条件是否成立，如果条件不成立，等待线程就要进入等待啦。

但是请注意！等待线程此时还持有“全局条件”的锁呢，如果这个锁不释放，那么唤醒线程该如何唤醒你呢，因此这个锁只能交给等待线程，在它进入等待前找到合适的时机释放，因此，这个锁需要传递给 pthread_cond_wait 函数，由该函数在何时的时机释放。如果我们把 pthread_cond_wait 函数展开，它大概会是这个样子的

这样，等该函数在适当的时机释放锁之后，唤醒线程便可以持有锁，然后对“全局条件”进行修改。

那么什么时候会出现“虚假唤醒”呢？这里举个例子，假设有三个线程 A、B、C，其中 A 和 B 是消费者，C 是生产者。

  • 线程 A 此时取走了生产线上的最后一个元素，然后在自己的 CPU 上去消费了
  • 线程 B 此时看到生产线上没有元素，于是乎调用了 pthread_cond_wait，把自己插入到等待队列中，并等待条件满足
  • 线程 C 此时向生产线上添加了一个新的元素，然后使用 pthread_signal 唤醒等待线程
  • 这个时候按理来说，线程 B 应该被唤醒，然后去消费生产线上的新元素了。
  • 但是非常有可能，在 B 元素从唤醒到获取锁的这段过程中，线程 A 完成了自己手头的消费，然后发现此时条件满足（指生产线上有东西），之后先于线程 B 获取了锁，线程 A 就跑去消费了。
  • 待线程 A 消费完，线程 B 终于拿到了锁，但此时实际上条件已经不满足了，因此需要 while 循环重新判断一下看条件是否满足。

以上就是 while 循环的作用了。
